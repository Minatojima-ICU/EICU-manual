# Webページ作成ガイド

このページは、[バイブコーディング練習ガイド（初心者向け）](./vibe_coding_preparation.md)と[ターミナルの練習](./terminal_practice.md)を修了していることを前提にしています。  
そのため、`cd` や `mkdir` などの基本操作、CLI型LLMの起動、簡単なプロンプトの書き方はすでにできるものとして進めます。

今回はGemini CLIを例に書きますが、CodexやClaude Codeでも流れはほぼ同じです。

## このページのゴール

- Webページ作成の全体の流れを理解する
- LLMに仕様を伝えてページを作らせる
- ローカルで確認しながら修正する
- GitHubに保存し、GitHub Pagesで公開する

## 全体の流れ

1. 作業用フォルダを作る
2. LLMに仕様を渡してページを作らせる
3. ローカルで動作確認する
4. GitHubにコードを保存する
5. GitHub Pagesで公開する
6. 修正して再公開する

## 事前に終わっている想定のこと

以下は、このページでは改めて詳しく説明しません。

- ターミナルを開ける
- ディレクトリを作成して移動できる
- Node.js / npm が使える
- Codex / Gemini CLI / Claude Code のいずれかを起動できる
- 短いプロンプトでファイル作成や修正を依頼できる

まだ不安がある場合は、先に以下を見直してください。

- [バイブコーディング練習ガイド（初心者向け）](./vibe_coding_preparation.md)
- [ターミナルの練習](./terminal_practice.md)

## 準備

### 1. GitHubアカウントを用意する

GitHub Pagesで公開するために、GitHubアカウントを使います。  
まだ持っていなければ作成してください。

### 2. GitHubでリポジトリを作る

ページ公開用のリポジトリを1つ作ります。

- リポジトリ名はわかりやすいものにする
- 通常は `Public` を選ぶ
- 初回は `README` を作っておいてもよい

### 3. 作業フォルダを作る

自分のPC内で、ページ作成用フォルダを作ります。  
例:

```bash
mkdir my-web-page
cd my-web-page
```

## まず理解しておくこと

### LLMは「実装担当」、自分は「仕様を決める人」

ページ作成で大事なのは、最初から完璧なコードを書くことではありません。  
最初にやるべきことは、何を作りたいかを言語化することです。

例えば以下を決めると、LLMに依頼しやすくなります。

- 何のためのページか
- 誰が使うか
- 何を表示するか
- 入力欄が必要か
- ボタンを押したら何が起きるか
- スマホでも見たいか
- GitHub Pagesで公開したいか

### 最初から細かすぎなくてよい

最初の指示は7割くらいで十分です。  
まず動くものを作り、その後で細かく修正していく方が進めやすいです。

## AGENTS.md / GEMINI.md を使う

### これは何か

`AGENTS.md` や `GEMINI.md` は、LLMに毎回伝えたい前提条件や作業ルールを書いておくためのファイルです。  
チャットのたびに同じ説明を繰り返さなくてよくなるため、ページ作成ではかなり便利です。

例えば以下のような内容を書きます。

- このプロジェクトは何を作るものか
- どの技術を使うか
- どのファイルを編集してよいか
- デザインや文体の方針
- GitHub Pagesで動く構成にしたいこと
- 日本語で説明してほしいこと

`AGENTS.md` と `GEMINI.md` のどちらを使うかは、使うツールや運用方針に合わせれば大丈夫です。  
大事なのは、プロジェクトの前提を1か所にまとめることです。

### AGENTS.md と GEMINI.md の違い

役割はほぼ同じで、どちらも「このプロジェクトではこう作業してほしい」という指示を書くファイルです。  
違いは主に、どのツール文化に合わせた名前か、という点です。

- `AGENTS.md`
  - より汎用的な名前です
  - Codex系の運用や、複数のLLMツールで共通利用したいときに向いています
  - 「AIエージェント向けの作業指示ファイル」と考えるとわかりやすいです
- `GEMINI.md`
  - Gemini CLIを中心に使うときにわかりやすい名前です
  - 「Geminiに読ませる前提ファイル」として運用できます

実際には、書く内容はほぼ同じで構いません。  
迷ったら、複数ツールで使い回しやすい `AGENTS.md` に寄せるのが無難です。

### どこに置けばいいか

基本は、**作業したいプロジェクトのルートディレクトリ直下** に置きます。  
つまり、`package.json` や `index.html`、`src` ディレクトリなどと同じ階層に置く、ということです。

例:

```text
my-web-page/
├─ AGENTS.md
├─ package.json
├─ index.html
├─ src/
└─ public/
```

あるいはGemini中心で運用するなら、次のようにします。

```text
my-web-page/
├─ GEMINI.md
├─ package.json
├─ index.html
├─ src/
└─ public/
```

この位置に置く理由は、LLMが「今作業しているプロジェクト全体のルール」として解釈しやすいからです。

### 1つだけ置けばよい

通常は `AGENTS.md` か `GEMINI.md` のどちらか1つで十分です。  
両方置くと、内容がずれたときに管理が面倒になります。

- Codexや複数ツールで使うなら `AGENTS.md`
- Gemini CLI中心なら `GEMINI.md`

という選び方で問題ありません。

### 基本構造

最初は、以下の4項目があれば十分です。

1. このプロジェクトの目的
2. 技術スタック
3. 作業ルール
4. 出力してほしい形式

### 書き方の例

```md
# AGENTS.md

## 目的
- 集中治療領域の論文を紹介するWebページを作る
- GitHub Pagesで公開する

## 技術スタック
- HTML / CSS / JavaScript を使う
- できるだけ構成はシンプルにする
- GitHub Pagesで動く静的サイトにする

## 作業ルール
- 初学者があとから読んで修正しやすい構成にする
- ファイルや関数の名前はわかりやすくする
- 大きく壊す変更をする前に、何を変えるか説明する
- 不明点があれば決め打ちせずに質問する

## デザイン方針
- スマホでも見やすくする
- 文字を小さくしすぎない
- 可読性を優先する

## 出力ルール
- 変更したファイル名を示す
- ローカル確認の方法を示す
- 必要なら次に実行するコマンドを書く
```

この程度でも、毎回のやり取りがかなり安定します。

### どう使うか

作業フォルダのルートに `AGENTS.md` または `GEMINI.md` を置いたうえで、LLMに以下のように伝えます。

```text
このディレクトリの AGENTS.md に従って作業してください。
まず内容を確認してから実装してください。
```

あるいは、最初の依頼文で次のように書いてもよいです。

```text
このプロジェクトでは AGENTS.md にルールを書いてあります。
その方針に沿って、最小構成で動くWebページを作ってください。
```

### 何を書くと効果が高いか

特に効果が高いのは以下です。

- GitHub Pagesで公開すること
- Reactではなく素のHTML/CSS/JavaScriptで作ること
- スマホ対応を必須にすること
- データを別ファイルに分けること
- 日本語で説明すること
- 大きな変更前には説明すること

### 最初から長く書きすぎなくてよい

`AGENTS.md` や `GEMINI.md` は、最初から完璧である必要はありません。  
実際に作業しながら、「毎回伝えていること」を少しずつ追加していけば十分です。

例えば次のように育てていきます。

- 最初は目的と技術スタックだけ書く
- 次にデザイン方針を追加する
- 次にファイル構成のルールを追加する
- 次に公開手順や確認手順を追加する

### 具体例

論文紹介ページを作るなら、例えば以下のような指示を書いておくと使いやすいです。

```md
## このプロジェクトで重視すること
- 論文データは追加しやすい形で管理する
- 一覧ページと詳細ページを分ける
- GitHub Pagesで壊れにくい構成にする
- スマホで読んでも見づらくならないようにする
- 医療者が読んで扱いやすい、落ち着いたUIにする
```

こうしておくと、毎回「スマホ対応で」「GitHub Pagesで」「一覧と詳細を分けて」と書かなくて済みます。

## 実際の作成手順

### 1. 作業フォルダでCLI型LLMを起動する

例:

```bash
gemini
```

Codexなら `codex`、Claude Codeなら `claude` です。

### 2. 最初の仕様を渡す

以下のように、課題・内容・条件を分けて渡すと伝わりやすくなります。

```text
## 課題
論文の批判的吟味をまとめたWebページを作りたい

## 内容
- 最初のページに論文一覧を表示する
- 各論文をクリックすると詳細ページに移動する
- 詳細ページには論文の要約、PICO、結果、考察、批判的吟味を表示する

## 条件
- GitHub Pagesで公開できる構成にする
- スマホでも見やすいデザインにする
- データは管理しやすい形で分離する
- 初学者でも後から修正しやすい構成にする
```

ここでは、まず大枠だけを伝えれば十分です。

### 3. 必要ならデータ構造も指定する

一覧ページや詳細ページを作る場合は、扱いたいデータの形も渡すと精度が上がります。

```text
各論文データには以下を含めてください。
- id
- 題名
- original_title
- 筆頭著者
- 雑誌名_巻号_出版年_ページ
- 登録日
- タグ
- 要約
- PICO
- 結果
- 考察
- 批判的吟味
- pmid
- pubmed_link
```

データ構造まで指定すると、後の修正がかなり楽になります。

### 4. 作業の進め方も指定する

LLMには、作るだけでなく進め方も指示できます。

例えば次のように言うと、不要な遠回りを減らせます。

```text
- まず最小構成で動くものを作ってください
- 作成したファイル構成を説明してください
- ローカルで確認する方法も教えてください
- 不明点があれば実装前に質問してください
```

## ローカルで確認する

コードができたら、必ず自分で表示を確認します。  
よくある確認コマンドは以下です。

```bash
npm install
npm start
```

プロジェクトによっては以下のこともあります。

```bash
npm run dev
```

どのコマンドを使うかわからなければ、LLMに「起動コマンドは何ですか」と聞いてください。

### 確認時に見るポイント

- エラーなく起動するか
- 文字化けしていないか
- ボタンやリンクが動くか
- スマホ幅でも大きく崩れないか
- 表示内容が仕様どおりか

## 修正の頼み方

うまくいかなかったときは、抽象的に「変です」と言うより、具体的に伝えた方が修正しやすくなります。

悪い例:

- レイアウトが変です

良い例:

- 論文一覧のカード同士の余白をもっと広げてください
- スマホ表示で文字が詰まるので、1列表示にしてください
- 詳細ページのPICOを表形式で表示してください
- PubMedへのリンクを目立つ位置に移してください

エラーが出たときは、エラーメッセージをそのまま貼って相談してください。

## GitHubに保存する

ローカル確認で問題なければ、GitHubに保存します。

```bash
git add .
git commit -m "first commit"
git push origin main
```

最初の接続設定がまだなら、LLMかGitHub公式手順を見ながらリポジトリを紐づけてください。

## GitHub Pagesで公開する

公開方法は構成によって少し変わります。  
そのため、ここでは「LLMに公開手順まで含めて作らせる」方針をおすすめします。

LLMへの追加指示例:

```text
このプロジェクトをGitHub Pagesで公開したいです。
必要な設定、package.jsonの修正、公開コマンドを整えてください。
その後、私が実行するコマンドも順番に示してください。
```

よく使う公開コマンドの例:

```bash
npm run deploy
```

公開後は、表示されたURLにアクセスして最終確認してください。

## 修正と再公開

ページ公開後も、修正は何度でもできます。  
基本の流れは毎回同じです。

1. LLMに修正を依頼する
2. ローカルで確認する
3. GitHubにpushする
4. 必要なら再deployする

この繰り返しで、少しずつ完成度を上げていきます。

## 実用的なプロンプト例

### 例1: 最初のページを作る

```text
## 課題
集中治療領域の論文を紹介するWebページを作りたいです。

## 内容
- トップページに論文一覧を表示してください
- 各論文をクリックすると詳細ページへ移動してください
- GitHub Pagesで公開できる静的サイトにしてください

## 条件
- 初学者でも後から修正しやすい構成にしてください
- スマホ表示にも対応してください
- データは別ファイルに切り出してください
- 最初はサンプル論文を3件入れてください
```

### 例2: デザイン修正を頼む

```text
現在のページを修正してください。

## 修正内容
- 全体の余白を少し広げる
- タグを色付きバッジで表示する
- 詳細ページの見出しを目立たせる
- スマホでは1カラム表示にする

## 条件
- 既存のデータ構造はできるだけ変えない
- 可読性を優先する
```

### 例3: 公開前チェックを頼む

```text
GitHub Pagesで公開する前に、問題になりそうな点を確認してください。
- 壊れたリンク
- パス指定の問題
- 画像やJSONの参照方法
- GitHub Pagesで動かない構成がないか
```

## 注意点

- 最初から大作を作ろうとしない
- まずは「表示できる最小構成」を作る
- 動いたら少しずつ機能追加する
- 1回の依頼で詰め込みすぎない
- 何を直したいかを具体的に言う
- 公開前に必ず自分で表示確認する

## うまくいかないとき

### 起動できない

- `package.json` があるか確認する
- `npm install` を先に実行する
- 起動コマンドが `npm start` ではなく `npm run dev` のこともある

### 公開したら表示が崩れる

- 相対パスの指定を確認する
- GitHub Pages用のベースパス設定が必要なことがある
- ローカルでは動いても公開環境で崩れることがあるので、URLで確認する

### 何を指示すればいいかわからない

LLMに次のように聞いて構いません。

```text
このWebページを作るために、最初に決めるべき仕様項目を箇条書きで整理してください。
```

## 次のステップ

- 実際に1つ、小さなページを作って公開する
- 公開したページの文言やデザインを修正する
- JSONやMarkdownからページ内容を生成する形に発展させる
- 論文紹介以外にも、計算機やチェックリストのページを作ってみる

このページの目的は、「Webページ作成をLLMと一緒に最後までやり切る」ことです。  
完璧な設計から始める必要はありません。まず動くものを作り、確認し、直し、公開する流れに慣れてください。
